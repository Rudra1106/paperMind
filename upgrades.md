- issues : 
Bug 1 (correctness, most serious): you're attaching completely unrelated papers as citations
Look at what got registered:
Registered citation index 7 in MEMORY: Scale-Specific Auxiliary Multi-Task Cont[rastive Learning for Deep Liver Vessel Segmentation]
Registered citation index 8 in MEMORY: Bridging Large Language Models and Single-Cell Transcriptomics...
Registered citation index 9 in MEMORY: Cooperative Assembly of Space Telescope Sub-Mirror Modules...
These are being attached as sources for a paper about VLMs playing action role-playing games. A liver vessel segmentation paper and a space telescope assembly paper have nothing to do with it. This means your enrichment pipeline is taking whatever top-1 result a search API returns — for a garbled query like "self_optimizable_action_generation_module" or "decomposable task-specific auxiliary modules" — and registering it as a real citation with zero relevance check. This is the exact failure mode Bohrium is designed to prevent (fabricated/irrelevant citation), and right now it's worse than showing no citation at all, because it actively misinforms.
Fix: add a relevance gate before any search result is accepted as a citation. Compute a simple similarity score between the query term and the returned title (token overlap via rapidfuzz.fuzz.token_set_ratio, or embedding cosine similarity if you want it stronger) and only register a citation above a threshold (start around 60-70). Below threshold, skip enrichment for that concept entirely rather than attaching a wrong source — an ungrounded concept card is honest; a wrongly-grounded one is not.
Bug 2 (root cause of the wrong citations and wasted time): you're routing paper-specific coined terms to public search APIs
Look at the actual query strings: varp_agent_framework, human_guided_trajectory_system, self_optimizable_action_generation_module, soag. These aren't general ML concepts like "attention mechanism" — they're this paper's own invented names for its own novel components (an ablation module, an internal system acronym). No external source will ever have a real match for these, by definition — they only exist in this one paper. Every one of these searches is guaranteed to return either nothing or garbage, and Bug 1 turns "nothing" into "garbage citation."
Fix — this is the actual efficiency win, not just a correctness one: add a classification step during extraction (extend your extraction prompt's category field beyond prerequisite/new to include a third value: paper_specific_term). Heuristics that catch most cases even before the LLM classifies: snake_case/CamelCase multi-word coinages, acronyms that only appear in this one paper, or terms the extraction agent itself generated by paraphrasing the author's terminology rather than lifting it verbatim. For anything tagged paper_specific_term, skip external enrichment entirely — its definition comes only from the paper's own evidence quote, nothing else. This alone likely removes half the external calls you're currently making, since a decent fraction of "concepts" in any paper are the authors' own named contributions, not textbook terms.
Bug 3 (the direct cause of slowness): no shared rate limiter, so every concept's backoff fires independently and simultaneously
Semantic Scholar rate limited (429). Retrying after 5.0s (attempt 1)...
Semantic Scholar rate limited (429). Retrying after 5.0s (attempt 1)...
Semantic Scholar rate limited (429). Retrying after 10.0s (attempt 2)...
Multiple concepts are each independently hitting 429, then each independently sleeping and retrying — nobody is coordinating to stay under the shared limit in the first place, so they all collide again on retry. Backoff-per-call without a shared limiter doesn't fix rate limiting, it just adds latency on top of still failing.
Fix: replace per-call backoff with a single shared token-bucket rate limiter (the aiolimiter package's AsyncLimiter, or a hand-rolled asyncio.Semaphore + delay) that every Semantic Scholar call goes through, sized to stay under ~1 request/3 seconds (comfortably under the unauthenticated 100-per-5-min pool). Concepts queue for a turn instead of all firing at once and all getting rejected together. Combined with Bug 2's fix (fewer calls made at all), this should cut your total enrichment time substantially.
Bug 4: connection pool exhaustion on Wikipedia
Connection pool is full, discarding connection: en.wikipedia.org. Connection pool size: 10
You're firing more concurrent Wikipedia requests than your HTTP client's pool allows, so connections get opened and immediately discarded — wasted work, not reused keep-alive connections. Fix: in wikipedia_client.py, explicitly configure httpx.Limits(max_connections=20, max_keepalive_connections=10) on a shared client instance (reused across requests, not recreated per-call), or route Wikipedia calls through the same semaphore as Semantic Scholar so concurrency never exceeds the pool size to begin with.

This fires on every single citation, all run. Nothing is actually persisting — you're building the References panel data in memory only, so it vanishes on restart and never survives a redeploy. Check your Supabase project's SQL editor / migrations folder — you likely wrote the CREATE TABLE citations (...) migration when we designed the citation registry, but it was never applied to the actual database. Run it, then confirm column names in the migration match exactly what your insert code sends (a mismatch here would throw a different error, but worth double-checking while you're in there).
